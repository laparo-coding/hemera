# Specs Style Guide

This repository uses English for all specification documents. The product UI remains German (DE).

## Language

- Specs (spec.md, plan.md, research.md, tasks.md, data-model.md, quickstart.md, contracts/\*):
  English
- UI copy in examples may include German labels (e.g., "Ausgebucht", "Jetzt buchen"). When used,
  provide an English gloss in parentheses at first mention.

## Structure per feature folder

Each `specs/###-slug/` folder should contain most of the following files:

- `spec.md` – functional and non-functional requirements, acceptance criteria, dependencies
- `plan.md` – implementation planning notes (planning only, no code changes)
- `research.md` – sources, findings, decisions, open questions
- `data-model.md` – entities and relationships (draft is fine)
- `quickstart.md` – how to preview and validate the feature locally
- `tasks.md` – generated by planning (if applicable)
- `contracts/` – OpenAPI schemas and contract tests (optional at first)

## Writing guidelines

- Focus on what and why; avoid implementation details in `spec.md`
- Make requirements testable and measurable
- Prefer consistent section titles across specs (as above)
- Use American English spelling

## Examples

- Acceptance scenario (UI label in German, gloss provided):
  - Given a course is sold out, Then I see a visible badge "Ausgebucht" (sold out) and the booking
    CTA is disabled.

## Review checklist

- [ ] Document is in English
- [ ] Section headers follow the structure
- [ ] Requirements are testable and unambiguous
- [ ] UI labels in German include an English gloss on first use
